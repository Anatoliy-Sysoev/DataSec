# Архитектурные решения

Документ фиксирует ключевые решения по DataSec-архитектуре. Формат близок к ADR, но упрощен для чтения в GitHub.

---

## ADR-001. Разделить аутентификацию и авторизацию

**Решение:**

- Keycloak отвечает за аутентификацию пользователя через SSO / OIDC.
- Apache Ranger отвечает за авторизацию доступа к данным в Trino.

**Обоснование:**

SSO подтверждает личность пользователя, но не решает вопрос доступа к catalog/schema/table/column. Для этого нужен отдельный слой политик доступа.

**Последствие:**

Приложение может доверять Keycloak для входа, но любой SQL-доступ должен проверяться через Ranger.

---

## ADR-002. Управлять доступами через IDM workflow

**Решение:**

Все регулярные доступы к данным должны запрашиваться и согласовываться через IDM-процесс.

**Обоснование:**

Service Desk фиксирует обращение, но не является полноценным механизмом жизненного цикла доступов. Для контролируемого отзыва доступа при переводе/увольнении нужна система, связанная с жизненным циклом пользователя.

**Последствие:**

Service Desk остается для исключений, инцидентов и аварийных операций. Типовые доступы должны идти через IDM.

## ADR-003. Хранить группы доступа в LDAP / FreeIPA / AD

**Решение:**

Группы доступа к данным хранятся в LDAP-совместимом каталоге. Членство в группах изменяется только через IDM-процесс.

**Обоснование:**

Ranger UserSync поддерживает синхронизацию пользователей и групп из LDAP/AD. Это стандартный и понятный способ передать групповые права в Ranger.

**Последствие:**

Запрещается ручное долгосрочное добавление пользователей в LDAP-группы вне процесса согласования.

---

## ADR-004. Писать политики Ranger на группы, а не на пользователей

**Решение:**

Политики Ranger создаются для LDAP-групп.

**Обоснование:**

Пользовательские политики не масштабируются и усложняют аудит. Групповая модель позволяет связать доступ с ролью, доменом данных и маршрутом согласования.

**Последствие:**

Исключения по отдельным пользователям должны быть временными и явно описанными.

---

## ADR-005. Использовать нативный Ranger access control в Trino

**Решение:**

Trino должен использовать `access-control.name=ranger`.

**Обоснование:**

В актуальных версиях Trino поддерживает нативную интеграцию с Apache Ranger. Это снижает потребность в сторонних нестабильных патчах.

**Последствие:**

Нужно контролировать совместимость версий Trino и Apache Ranger.

---

## ADR-006. Доставлять audit-события в SIEM

**Решение:**

Audit-события Ranger должны передаваться в централизованный контур мониторинга безопасности.

**Обоснование:**

Локальный аудит не решает задачи ИБ: расследования, корреляции, выявления массового чтения, access denied burst и доступа к чувствительным данным.

**Последствие:**

Минимальный поток: Ranger Audit -> OpenSearch/Solr/Log shipper -> SIEM.

---

## ADR-007. Вести реестр Trino/Ranger-сервисов

**Решение:**

Каждый Trino/Ranger-контур должен быть внесен в реестр.

**Минимальные поля реестра:**

- проект / домен данных;
- URL Trino;
- URL Ranger Admin;
- Ranger service name;
- владелец;
- команда сопровождения;
- audit destination;
- статус SIEM-интеграции;
- статус LDAP/IDM-интеграции;
- SLA / критичность.

**Обоснование:**

Без реестра невозможно централизованно проводить аудит и контролировать покрытие платформы.
