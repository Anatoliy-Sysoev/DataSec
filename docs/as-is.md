# AS IS-архитектура

Страница описывает типовую разрозненную модель доступа к данным, которую целевая DataSec-архитектура должна заменить.

## Текущее состояние

```text
Пользователь -> заявка в Service Desk -> ручное согласование -> LDAP-администратор вручную добавляет пользователя в группу
Пользователь -> Keycloak SSO -> Trino
LDAP -> Ranger UserSync -> политики Ranger -> Ranger plugin в Trino
Ranger audit -> локальное audit-хранилище / частичная выгрузка
```

## Характеристики

| Область | AS IS-поведение | Риск |
|---|---|---|
| Запрос доступа | Заявка в Service Desk | Нет структурированного каталога доступов |
| Исполнение | Ручное изменение LDAP-группы | Ошибки, задержки, зависимость от исполнителя |
| Отзыв доступа | Ручной или запаздывающий | Избыточный доступ после перевода/увольнения |
| SSO | Keycloak используется для входа | SSO не равен авторизации к данным |
| Авторизация | Ranger может использовать LDAP-группы | Качество зависит от ручной гигиены групп |
| Аудит | Локальное хранение или неполная выгрузка | Слабая наблюдаемость для ИБ/SOC |
| Governance | Нет единого реестра Ranger/Trino | Сложный аудит по проектам |

## Основные проблемы

1. Ручное изменение LDAP-групп не масштабируется.
2. Service Desk не является полноценной системой управления жизненным циклом доступов.
3. Keycloak-аутентификация и LDAP/Ranger-авторизация могут рассинхронизироваться.
4. Политики Ranger трудно аудировать без стандарта именования и владельцев.
5. Командам ИБ нужен API-based аудит и реестр, а не ручная проверка UI.
6. Перевод и увольнение сотрудника должны автоматически отзывать доступ.
