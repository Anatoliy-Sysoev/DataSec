# Интеграция с IDM

## Назначение

IDM — целевая точка управления жизненным циклом доступов. Пользователь не должен получать доступ к данным через ручное добавление в LDAP-группу. Вместо этого он запрашивает доступ в IDM, проходит согласование, а IDM меняет членство в LDAP-группах.

---

## Целевой процесс

```text
HR / источник сотрудников
  -> IDM
  -> заявка на доступ
  -> согласование
  -> изменение членства в LDAP-группе
  -> Ranger UserSync
  -> политики Ranger
  -> доступ в Trino
```

---

## Объекты IDM

| Объект | Описание |
|---|---|
| Система | Trino Data Platform / конкретный Trino-контур |
| Роль | Бизнес-роль доступа к данным |
| LDAP-группа | Техническая группа, используемая Ranger |
| Владелец данных | Ответственный за согласование доступа к домену данных |
| Согласующий ИБ | Дополнительное согласование для чувствительных данных |
| Срок доступа | Постоянный или временный доступ |

---

## Пример маппинга роли на LDAP-группу

| IDM-система | IDM-роль | LDAP-группа | Ranger policy |
|---|---|---|---|
| Trino Data Platform | Продажи: чтение заказов | `dl_sales_orders_ro` | `trino_sales_orders_select` |
| Trino Data Platform | Продажи: ПДн, маскирование | `dl_sales_pii_masked` | `trino_sales_pii_masked_select` |
| Trino Data Platform | Финансы: чтение платежей | `dl_finance_payments_ro` | `trino_finance_payments_select` |

---

## События жизненного цикла

| Событие | Действие IDM |
|---|---|
| Прием сотрудника | Создать учетную запись / разрешить запрос доступов |
| Перевод сотрудника | Пересчитать доступы, отозвать нерелевантные группы |
| Увольнение | Удалить из LDAP-групп доступа, заблокировать учетную запись |
| Временный доступ истек | Удалить пользователя из соответствующей LDAP-группы |

---

## Требования к API / интеграции

IDM должен уметь:

1. Отобразить пользователю понятный каталог доступов.
2. Запустить маршрут согласования.
3. Добавить пользователя в LDAP-группу после согласования.
4. Удалить пользователя из LDAP-группы при отзыве доступа.
5. Хранить историю заявки и согласований.
6. Передавать события в журнал аудита.
7. Предоставлять выгрузку ожидаемого состояния доступов для reconciliation.

---

## Что не должно быть целевым процессом

- Ручное добавление пользователя в LDAP-группу по обычной заявке.
- Ручное создание постоянных политик Ranger на пользователя.
- Хранение фактического доступа только в комментариях Service Desk.
- Отсутствие владельца у роли доступа.

---

## Контрольная сверка

Регулярно должна выполняться сверка:

```text
IDM expected access
  vs
LDAP actual group membership
  vs
Ranger actual policies
  vs
Ranger audit actual usage
```

Результат сверки должен формировать отчет о расхождениях.
